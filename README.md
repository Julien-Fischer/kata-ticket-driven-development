# Ticket-Driven Design (Kata)

FeatureFactory GmbH wants to update their pricing engine to increase ticket sales through its website.

One small hitch: the pace of delivery has slowed to a crawl. Nobody is quite sure why.
After months of sluggish output, the CTO brought you in to figure out what went wrong and get things back on track.

Your review uncovered a frantic, ticket-driven culture where features were rushed out with zero modeling effort.
As a result, the product deteriorated to the point where even minor changes could trigger failures anywhere.

To restore confidence and stability, you introduced tests that captured existing behavior,
creating a foundation for safer, more deliberate refactoring.

Now it’s time to rescue this product from spaghetti hell, before implementing the next big feature.

## Goal

The problem that this code is designed to solve is explained here: [Ticket-Driven Development](https://craft.coach/kata/ticket-driven-development)

## Running tests

```bash
npm install       
``` 
Then either run
```bash
npm run test
```
or run instead:
```bash
npm run test:watch 
```
